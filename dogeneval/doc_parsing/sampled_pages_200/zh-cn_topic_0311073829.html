扩展QCI/5QI应用

表1 不同场景中UNC对QCI/5QI的处理方式

场景

QCI来源

相关流程

场景1

签约QoS

UNC从UDM获取签约QoS

场景2

动态PCC规则

UNC从PCF获取动态PCC规则（Npcf_SMPolicyControl_Create Response）

场景3

签约QoS
  • 用户附着，UNC从MME收到创建会话的请求（Create Session Request）
  • UNC从MME收到更新缺省承载的请求（Modify Bearer Command）

场景4

动态PCC规则
  • UNC从PCF/PCRF获取动态PCC创建缺省承载/专有承载
  • 动态PCC下，PCF/PCRF主动更新用户缺省承载

场景5

动态PCC规则

对于GPRS/UMTS用户，一次/二次PDP上下文激活/更新流程中，UNC收到PCRF下发的更新PDP消息/CCA应答消息。
图1 场景1：UNC从UDM获取签约QoS
  1. 5QI如果不在1~255之间，会被UNC认定为不合法，则通过SET 5GCQOSCORRT进行纠错。
  2. 当5QI为扩展5QI时，需要打开本特性的License，未开启License则使用SET 5GCQOSCORRT配置的5QI进行激活。
  3. 如果是本地用户，则使用该扩展5QI进行接入。针对漫游用户，还需要开启SET EXTENQCIATTR中的“QCICTRLFLAG（用户接入的扩展QCI控制开关）”。

    • 当开关开启，则根据ADD EXTENDQCIMAP配置扩展5QI和标准5QI的映射关系，将扩展5QI转换成标准5QI进行接入，未配置ADD EXTENDQCIMAP时使用 SET EXTENQCIATTR配置的缺省映射值。

      UNC漫游用户使用扩展QCI接入时，还可以通过软参BIT_1858 控制漫游用户使用扩展QCI接入时UNC是否将扩展QCI按照本地配置映射成为标准的QCI处理。

      • 值为0时，按照本地配置映射成标准QCI处理。
      • 值为1时，不按照本地配置映射。
    • 当开关关闭，则拒绝用户接入。

图2 场景2：UNC从PCF获取动态PCC规则
  1. 5QI如果不在1~255之间，会被UNC认定为不合法，则通过SET 5GCQOSCORRT进行纠错。
  2. 当5QI为扩展5QI时，需要打开本特性的License，未开启License则拒绝用户接入。
  3. 根据不同用户进行验证：

    • 如果是本地用户，至少满足以下条件之一才允许用户使用该扩展5QI接入，否则拒绝：
      • UNC对接的Ga接口/Gy接口/Radius鉴权和计费/N40接口都支持扩展5QI（可通过SET EXTENQCIATTR配置开关）。
      • 根据SET EXTENQCIATTR或ADD EXTENDQCIMAP配置的映射关系将扩展5QI映射为标准5QI，验证PCF下发的资源类型是否与ADD STDQOSID中该标准5QI配置的资源类型一致。
    • 针对漫游用户，还需要开启SET EXTENQCIATTR中的“QCICTRLFLAG（用户接入的扩展QCI控制开关）”。
      • 当开关开启，则根据ADD EXTENDQCIMAP配置扩展5QI和标准5QI的映射关系，将扩展5QI转换成标准5QI进行接入，未配置ADD EXTENDQCIMAP时使用 SET EXTENQCIATTR配置的缺省映射值。

        UNC漫游用户使用扩展QCI接入时，还可以通过软参BIT_1858 控制漫游用户使用扩展QCI接入时UNC是否将扩展QCI按照本地配置映射成为标准的QCI处理。

        • 值为0时，按照本地配置映射成标准QCI处理。
        • 值为1时，不按照本地配置映射。
      • 当开关关闭，则拒绝用户接入。

图3 场景3：UNC收到MME创建会话的请求（Create Session Request）/更新缺省承载的请求（Modify Bearer Command）
  1. QCI如果不在1~255之间，会被UNC认定为不合法，则通过SET DEFEPSQOS进行纠错。
  2. 根据QCI的类型进行判断：

    • 当QCI为标准QCI时，如果QCI值为65/66/69/70时，需要开启WSFD-102602 LTE一键通（适用于PGW-C/SGW-C）功能，否则拒绝用户接入。
    • 当QCI为扩展QCI时，需要打开本特性的License，未开启License则使用SET DEFEPSQOS配置的QCI进行激活。

      SGW-C和PGW-C分离场景，可能存在PGW-C有扩展QCI的license但是SGW-C无扩展QCI的license的情况。此情况下SGW-C收到携带扩展QCI的专有承载创建请求时的处理受软参控制：

      BIT844 控制SGW-C在无扩展QCI的license时，收到携带扩展QCI的专有承载激活请求是否允许激活成功。

      • 取值为0时，允许激活成功，使用标准QCI。
      • 取值为1时，不允许激活。

  3. 如果是本地用户，则使用该扩展QCI进行接入。针对漫游用户，还需要开启SET EXTENQCIATTR中的“QCICTRLFLAG（用户接入的扩展QCI控制开关）”。

    • 当开关开启，则根据ADD EXTENDQCIMAP配置扩展QCI和标准QCI的映射关系，将扩展QCI转换成标准QCI进行接入，未配置ADD EXTENDQCIMAP时使用 SET EXTENQCIATTR配置的缺省映射值。

      UNC漫游用户使用扩展QCI接入时，还可以通过软参BIT_1858 控制漫游用户使用扩展QCI接入时UNC是否将扩展QCI按照本地配置映射成为标准的QCI处理。

      • 值为0时，按照本地配置映射成标准QCI处理。
      • 值为1时，不按照本地配置映射。
    • 当开关关闭，则拒绝用户接入。
